# v2 Roadmap

> Status tracker for post-v1 features. Items are ordered by priority within tiers.
> Each item includes pointers to upstream references, relevant OCaml modules, and
> implementation notes to make pickup straightforward.

---

## Completed

### 1. Output API (structured output / schema validation)

**Status:** Done (PR #4, merged to main)

Implemented `Output.text`, `Output.object_`, `Output.enum`, `Output.array`,
`Output.choice` with JSON Schema validation via the `jsonschema` opam package.
Wired through `generate_text`, `stream_text`, and `server_handler`.

**Files:** `lib/ai_core/output.ml`, `lib/ai_core/partial_json.ml`
**Plan:** `docs/plans/2026-03-25-output-api.md`

### 2. v6-only request parsing (`convertToModelMessages`)

**Status:** Done

Rewrote `server_handler.ml` to parse all v6 UIMessage part types:
- **Text**, **file** (url + base64), **reasoning** for user and assistant messages
- **Tool invocation** parts (`tool-{name}` and `dynamic-tool`) with full state
  machine: `input-streaming`, `input-available`, `output-available`,
  `output-error`, `output-denied`, `approval-requested`, `approval-responded`
- Tool invocations produce both `Assistant` (with `Tool_call` parts) and
  `Tool` (with `tool_result` entries) messages, matching upstream
  `convertToModelMessages` behavior

Removed v5 `content` string fallback. Replaced melange-json PPX parsing with
manual Yojson parsing (melange-json PPX cannot derive `Yojson.Basic.t option`).
Exposed `parse_messages_from_body` in `.mli` for testing. 39 tests covering
all part types, tool states, edge cases, and malformed input.

**Files:** `lib/ai_core/server_handler.ml`, `test/ai_core/test_server_handler.ml`

### 3. UIMessage stream composition (`createUIMessageStream` / `createUIMessageStreamResponse`)

**Status:** Done

Implemented `Ui_message_stream_writer` module with composable stream builder API:
- Opaque writer type with `write` (synchronous) and `merge` (non-blocking via
  `Lwt.async` with safety guarantees)
- `create_ui_message_stream` manages lifecycle (Start/Finish/Error chunks),
  tracks in-flight merges via ref-counting, supports `on_finish` callback
- `create_ui_message_stream_response` wraps chunk stream as HTTP SSE response
  with CORS headers
- Minimal `on_finish` with `finish_reason` and `is_aborted` (full message
  reconstruction deferred to item #13)
- Explicit `?message_id` instead of `original_messages` for persistence

**Files:** `lib/ai_core/ui_message_stream_writer.ml`, `test/ai_core/test_ui_message_stream_writer.ml`
**Design:** `docs/plans/2026-03-27-ui-message-stream-composition-design.md`

### 4. Tool approval workflow

**Status:** Done

Implemented stateless tool approval matching upstream AI SDK's `needsApproval` pattern:
- `Core_tool.t` gains `needs_approval : (Yojson.Basic.t -> bool Lwt.t) option` with
  `create` and `create_with_approval` constructors
- Step loops in `generate_text` and `stream_text` partition tool calls: if any tool
  needs approval, the loop stops and emits `Tool_approval_request` chunks
- `server_handler` parses `approval-responded` parts with `approved` field, collects
  pre-approved tool call IDs, and threads them through `~approved_tool_call_ids` so
  the step loop skips re-checking on re-submission
- New `Tool_approval_request` chunk type in `Ui_message_chunk` serialized as
  `tool-approval-request` in SSE stream

**Files:** `lib/ai_core/core_tool.ml`, `lib/ai_core/generate_text.ml`,
`lib/ai_core/stream_text.ml`, `lib/ai_core/stream_text_result.ml`,
`lib/ai_core/server_handler.ml`, `lib/ai_core/ui_message_chunk.ml`,
`lib/ai_core/text_stream_part.ml`
**Plan:** `docs/plans/2026-03-27-tool-approval-workflow.md`

## High Priority

### 5. Cross-language interop test suite

**What:** Node.js script using `readUIMessageStream` from `ai@6` to consume
our SSE output and verify it parses correctly. Automated CI test.

**Current state:** OCaml-side unit tests exist for chunk serialization. No
end-to-end test that proves a real `useChat()` client can consume our output.

**Approach:**
- Node.js test script in `test/interop/` that starts the OCaml chat server,
  sends requests, pipes responses through `readUIMessageStream`, and asserts
  on the parsed messages.
- Derive test cases from upstream AI SDK test suite where possible.
- Cover: basic text streaming, tool calls, reasoning, structured output,
  error responses, CORS headers.

**Key files:**
- New: `test/interop/test_stream_compat.mjs`
- Reference: `examples/chat_server/` for server setup patterns

**Upstream reference:**
- https://ai-sdk.dev/docs/ai-sdk-ui/reading-ui-message-streams

**Depends on:** Nothing

## Medium Priority

### 6. `smoothStream` text buffering

**What:** Buffer text deltas and emit them at a controlled rate for smoother
UI rendering. Configurable by word boundary, delay, and chunk size.

**Current state:** Text deltas are forwarded as-is from the provider. Some
models emit very small or very large chunks that feel janky in the UI.

**Design:** A stream transformer `Ui_message_chunk.t Lwt_stream.t ->
Ui_message_chunk.t Lwt_stream.t` that buffers `Text_delta` chunks and
re-emits them word-by-word or on a timer.

**Upstream reference:** https://ai-sdk.dev/docs/ai-sdk-core/streaming-text#smooth-text-streaming

**Depends on:** Nothing

**Upstream reference:** https://ai-sdk.dev/docs/ai-sdk-core/telemetry

### 8. `stopWhen` predicate for step loop

**What:** Currently only `max_steps` limits the tool loop. Add a
`?stop_when:(Generate_text_result.step -> bool)` parameter for dynamic
termination matching the TypeScript SDK's `stopWhen`.

**Key files:**
- `lib/ai_core/generate_text.ml` — add parameter to step loop
- `lib/ai_core/stream_text.ml` — same
- `lib/ai_core/server_handler.ml` — optionally expose

**Upstream reference:** https://ai-sdk.dev/docs/ai-sdk-core/generating-text

**Depends on:** Nothing

## Low Priority

### 10. `transient` field on custom `Data` chunks

**What:** The v6 spec supports `transient: true` on `data-*` chunks.
Transient data is sent to the client but not persisted in message history
(useful for progress indicators, notifications).

**Current state:** `Ui_message_chunk.Data` has `data_type`, `id`, `data`
but no `transient` field.

**Key files:**
- `lib/ai_core/ui_message_chunk.ml` — add `?transient:bool` to `Data`

**Depends on:** UIMessage stream composition (#3) for ergonomic usage

### 13. Full `onFinish` callback with message reconstruction

**What:** Enhance `create_ui_message_stream`'s `on_finish` callback to provide
the accumulated response message (`messages`, `responseMessage`,
`isContinuation`) and accept `original_messages` for automatic continuation
detection and message ID assignment. Requires building a chunk accumulator
that reconstructs a UIMessage from the stream.

**Current state:** The v1 implementation (#3) provides a minimal `on_finish`
with `finish_reason` and `is_aborted` only, plus explicit `?message_id`.
This was a deliberate scoping decision — the stream composer is a generic
primitive not tied to LLM output, and `stream_text_result.t` already provides
persistence data when an LLM call is involved.

**Key files:**
- `lib/ai_core/ui_message_stream_writer.ml` — extend `on_finish`, add
  `?original_messages` parameter, build chunk accumulator
- New: `lib/ai_core/ui_message_accumulator.ml` — reconstruct UIMessage from
  chunk stream

**Upstream reference:** https://ai-sdk.dev/docs/reference/ai-sdk-ui/create-ui-message-stream

**Depends on:** UIMessage stream composition (#3)

---

### 14. `ai_sdk_react` bindings parity

**What:** Keep the Melange `ai_sdk_react` bindings in sync with upstream
`@ai-sdk/react` types as new features land in the OCaml SDK.

**Current state:** Bindings cover `useChat`, `useCompletion`, all message
part types including the newly added `data_ui_part` (`DataUIPart`).

**Done so far:**
- `data_ui_part` type with `data`, `id`, `data_type` accessors
- `Data of data_ui_part` variant in `classified_part`
- `classify` matches `data-*` prefixed types
- `custom_stream` example demonstrates data parts in Melange frontend

**Remaining (add as upstream features are implemented):**
- `transient` field on data parts (when #10 lands)
- Step finish callback bindings (when tool approval #4 lands)
- Source parts rendering helpers
- `onData` typed callback (currently bound as `Js.Json.t -> unit`)

**Key files:**
- `bindings/ai-sdk-react/types.ml` / `.mli` — part types and classify
- `bindings/ai-sdk-react/use_chat.ml` / `.mli` — hook bindings

**Depends on:** New features landing that add chunk/part types
